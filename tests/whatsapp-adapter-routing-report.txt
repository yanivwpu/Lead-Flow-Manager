========================================
     ARCHITECTURE ANALYSIS
========================================

OUTBOUND MESSAGES (user sends to contact):
  → channelService.sendMessage() → WhatsAppAdapter.send()
  → Adapter checks user.whatsappProvider
  → Routes to Meta Graph API OR Twilio API
  → Provider decision is per-user, not per-conversation

INCOMING MESSAGES (contact replies):
  → Meta:   POST /api/webhook/meta → parseMetaIncomingWebhook → addInboxJob
  → Twilio: POST /api/webhook/twilio/incoming → parseIncomingWebhook → addInboxJob
  → These are SEPARATE endpoints called by the provider directly
  → No provider routing needed — the webhook URL determines the path

DELIVERY STATUS UPDATES:
  → Meta:   Handled in POST /api/webhook/meta via parseMetaStatusWebhook
  → Twilio: Handled in POST /api/webhook/twilio/status via parseStatusWebhook
  → Each provider reports status through its own webhook

AUTO-REPLIES:
  → Meta webhook:   Uses sendMetaWhatsAppMessage (correct)
  → Twilio webhook: Uses sendUserWhatsAppMessage (correct)
  → Each path uses the same API that delivered the incoming message

OLD CONVERSATIONS/CONTACTS:
  → Routing is based on user.whatsappProvider, NOT conversation metadata
  → Old contacts/conversations route to whichever provider is currently active
  → Verified in Test 8 with a pre-fix legacy contact

VALIDATION METHOD:
  → MOCKS ONLY — no live Meta or Twilio sandbox was used
  → global.fetch was intercepted to simulate Meta Graph API responses
  → Twilio client creation was validated via provider path logic
  → Code path analysis (Test 9) verified webhook handler source code
  → For production confidence, a Meta test phone number or Twilio test
    credentials should be used in a staging environment
